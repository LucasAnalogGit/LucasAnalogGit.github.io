# Python 读取仿真结果并提取模拟电路性能指标

## 1. 背景：为什么需要自动提取性能指标

前三篇文章依次解决了三个问题：如何通过 virtuoso-bridge 让 Python / AI Agent 连接 Cadence Virtuoso，如何使用 SKILL 自动生成模拟电路原理图，以及如何构建基于 Spectre/OCEAN 的可重复自动仿真流程。完成这三步之后，自动化系统已经可以根据一组设计参数生成电路、运行仿真并把结果写入独立的 trial 目录。

但到这里，自动化链路只走完了一半。Spectre 输出的是波形、工作点和日志文件，而优化算法无法直接消费一条 AC 幅频曲线或一段 transient 波形。优化器需要的是可计算、可比较、可排序的标量，例如 DC gain、GBW、PM、SR 和 power。把波形转换为这些标量的过程，就是本文要讨论的性能指标提取层。

这里需要先强调一个容易被忽略的判断：

> 自动仿真本身并不等价于自动设计。只有将仿真波形转化为可计算、可比较、可优化的性能指标，优化算法才能真正参与模拟电路设计。

换句话说，一个能稳定 run 仿真的流程，离“能自动设计电路”还差一个关键模块：把仿真结果翻译成结构化指标，并判断这些指标是否可信。本文关注的就是这一层在整体流程中的位置：

```text
Spectre 仿真结果
        ↓
Python 读取波形 / 日志 / 数据文件
        ↓
提取 Gain、GBW、PM、SR、Power 等性能指标
        ↓
生成结构化 metrics
        ↓
反馈给优化器
```

手动设计时，工程师在 ADE 或 Maestro 中点开波形、放置 marker、读取数值，凭经验判断曲线是否正常。这种判断在自动化流程中必须被代码化：哪些信号要读、用什么数学定义提取、什么情况下结果无效、提取失败如何上报，都需要明确写下来。否则优化器会把仿真异常当作电路性能差，或者把无效数据用于下一轮采样。

## 2. Spectre 仿真结果的常见形式

在读取数据之前，需要先弄清 Spectre 到底产生了哪些文件，以及它们各自的作用。

### 2.1 PSF 与 PSFASCII

PSF（Parameter Storage Format）是 Cadence 仿真结果的原生格式，用于保存各 analysis（dc、ac、tran、stb 等）的波形和工作点数据。它有两种常见形态：

- **二进制 PSF**：体积小、读写快，是大规模仿真的默认选择，但需要专门的解析库或通过 OCEAN/结果访问 API 读取；
- **PSFASCII**：以文本形式保存同样的数据，便于直接查看和调试，代价是文件更大、读取更慢。

在自动化流程中，是否使用 `psfascii` 取决于读取方式和数据规模。如果用 OCEAN 表达式直接在结果目录上计算指标，二进制 PSF 通常足够；如果希望用 Python 直接解析文本、不依赖 Cadence 进程，PSFASCII 会更方便调试。两者可以共存：内循环用二进制 PSF 求速度，少量需要复核的样本再导出 ASCII。

无论哪种格式，关键是**读取脚本要与所约定的格式匹配**。结果目录里有数据，不等于当前脚本能打开它。

### 2.2 Spectre 日志文件

`spectre.out`（或等价的仿真日志）记录了 netlist 解析、analysis 执行、收敛行为和 error/warning 信息。日志文件不直接提供性能指标，但它是判断一次仿真是否值得提取指标的第一道依据：

- netlist 是否语法正确；
- model file / section 是否成功 include；
- DC operating point 是否收敛；
- 各 analysis 是否正常结束；
- 是否出现 `error`、`fatal` 或关键 `warning`。

在自动化流程里，日志应当被保留并做最小化解析。如果日志里已经报告 DC 不收敛，那么后续的 AC gain 即使能算出一个数字，也没有物理意义。

### 2.3 CSV / JSON 中间结果

PSF 是仿真器的原生输出，但它不是优化器的理想输入。工程上通常会在 PSF 与优化器之间插入一层**中间结果文件**：

- 用一个文本文件（例如 `performance.txt`）承接 OCEAN/表达式计算出的原始指标；
- 用 `metrics.json` 保存结构化、带单位、带有效性标记的指标；
- 用 `summary.csv` 汇总所有 trial 的关键指标，便于排序和绘图。

这一层的价值在于解耦：PSF 属于仿真器世界，JSON/CSV 属于优化器世界。优化器不需要知道波形是怎么存的，只需要读一个稳定的结构化文件。

### 2.4 为什么需要结构化 metrics

把上面三类文件串起来，可以得到一条清晰的数据流：

```text
波形 (PSF/PSFASCII)  →  表达式/Python 计算  →  原始指标 (performance.txt)
                                              ↓
                                   结构化指标 (metrics.json)
                                              ↓
                                          优化器
```

结构化 metrics 是这条链路的产物，也是后续所有环节的输入。它让指标可以被程序读取、跨 trial 比较、批量绘图、筛选可行设计，并在出错时快速 debug。本文后续讨论的所有提取方法，最终都要落到这样一个结构化结果上。

## 3. 从波形到指标的基本思想

### 3.1 波形数据的数学表示

从数据角度看，任何仿真波形都是离散采样序列。一条时域输出可以写成：

```text
y = {y(t_0), y(t_1), ..., y(t_N)}
```

对于 AC 小信号仿真，结果是复数频率响应，可在每个频点上写成幅值与相位：

```text
H(f) = |H(f)| · e^{jφ(f)}
```

指标提取的本质，就是把这些离散序列映射为一组标量：

```text
metrics = Extract(waveforms)
```

`Extract` 是一组确定性函数。理解这一点很重要：指标提取不是“看波形”，而是“在离散数据上执行明确定义的数学运算”。每个性能指标对应一个可以写成公式的算子。

### 3.2 采样点、插值与单位换算

离散采样带来三个必须处理的工程细节：

1. **频点/时间点是有限且离散的**。我们关心的特征点（例如增益穿越 0 dB 的频率）几乎不会正好落在某个采样点上，需要**插值**。AC 仿真的频率轴通常是对数刻度，插值时应在对数频率上进行，否则会引入明显误差。
2. **单位需要统一**。Spectre 内部用 SI 单位（V、A、Hz、s），但工程报告习惯用 dB、V/µs、mW 等。提取层应当在一个固定的位置完成单位换算，避免同一个量在不同函数里用不同单位。
3. **采样密度影响精度**。slew rate 这类依赖导数的指标对时间步长敏感；GBW 这类依赖穿越点的指标对频点密度敏感。提取算法的鲁棒性，部分取决于仿真设置是否给出了足够的采样。

### 3.3 指标提取与异常判断

自动化与手动读数最大的区别在于：手动读数时人会顺手判断“这条曲线是不是正常的”，而自动化必须把这种判断写成代码。因此每个指标提取函数实际上承担两个任务：

- **计算数值**：在合法波形上算出标量；
- **判断有效性**：识别曲线缺失、未穿越、未收敛、NaN/Inf 等异常，并明确返回“无效”，而不是返回一个看似合理却没有物理意义的数字。

把有效性判断前置到提取层，是整个优化闭环稳定的前提。这一点会在第 5、7 节反复出现。

## 4. 常见模拟电路性能指标的提取方法

下面按指标逐一给出数学定义和提取要点。这里只讨论方法，不绑定任何具体 testbench 实现。

### 4.1 DC Gain

对 AC 小信号分析，低频增益取传递函数在低频点的幅值：

```text
A_0 = |H(f_low)|
```

以 dB 表示：

```text
Gain_dB = 20 · log10(|A_0|)
```

要点：

- 对单端输出，`H(f)` 可取 `Vout / Vin`；
- 对全差分电路，应使用差分定义 `Vout_diff / Vin_diff`，即 `(Voutp − Voutn) / (Vinp − Vinn)`，否则增益会偏差一个因子甚至失去物理意义；
- `f_low` 必须落在平坦的低频段，不能取在异常点或未收敛点上；
- 单位与差分定义要在整个流程中保持一致。

### 4.2 GBW / UGB

单位增益带宽是增益降到 1（即 0 dB）的频率：

```text
|H(f_ugb)| = 1   ⇔   Gain_dB(f_ugb) = 0 dB
```

由于频点离散，通常需要在 0 dB 处对频率插值：

```text
f_ugb ≈ interp(Gain_dB(f), 0 dB)
```

要点：

- 频率轴用对数坐标，插值在 log10(f) 上做；
- 直接取“最接近 0 dB 的采样点”会带来不可忽略的误差，应在相邻两点之间线性插值；
- 如果增益曲线从未穿越 0 dB（例如增益太低或仿真异常），该指标无效，应明确返回无效，而不是返回端点频率。

### 4.3 Phase Margin

相位裕度在单位增益频率处计算：

```text
PM = 180° + phase(f_ugb)
```

要点：

- 这里采用常见的负反馈环路增益相位约定；不同 testbench 或 stb 分析下，相位定义和符号可能不同；
- 自动提取时必须统一**相位展开（phase unwrap）方式**，否则 ±360° 的跳变会让 PM 完全错误；
- PM 依赖 `f_ugb`，所以 GBW 提取失败时 PM 也应判为无效；
- 简单 OTA 可用差分输出传递函数估计，复杂反馈或 CMFB 结构通常需要专门的 stb 环路分析，提取公式要与之对应。

### 4.4 Slew Rate

大信号压摆率是输出电压随时间变化率的最大值：

```text
SR = max |dVout/dt|
```

离散数据上用差分近似：

```text
SR ≈ max |(Vout[i+1] − Vout[i]) / (t[i+1] − t[i])|
```

要点：

- 应区分 rising slew rate 与 falling slew rate，二者物理含义不同；
- 不能把噪声尖峰或数值毛刺当成有效 SR，通常需要在指定的时间窗口内计算，并对波形做必要的限幅/裁剪；
- 注意单位换算，例如从 V/s 转为 V/µs；
- 时间窗口选取要覆盖真实的转换沿，又要避开后续的振铃和 settling 段。

### 4.5 Power

平均功耗由电源电压与电源电流得到：

```text
P = VDD · I_avg
```

对瞬态仿真，平均电流是电源电流在一个周期内的时间平均：

```text
I_avg = (1/T) ∫ i_DD(t) dt   ≈   mean(i_DD)
```

要点：

- Spectre 中电源支路电流的方向可能与直觉相反，提取时要统一符号约定（通常取绝对值或固定参考方向）；
- 静态功耗可直接用 DC 工作点的电源电流，动态/平均功耗用 transient 的电流积分；
- 多电源系统（例如模拟电源、数字电源、衬底偏置）应分别统计再求和。

### 4.6 Output Swing、Settling Time 与其他指标

除上述五个核心指标外，常见的还有：

- **Output Swing**：在大信号或扫描下输出能达到的有效摆幅，需排除进入饱和或截止的区段；
- **Settling Time**：阶跃响应进入并保持在目标误差带（例如 ±0.1%）内所需的时间，依赖误差带定义和稳定判据；
- **CMRR / PSRR / Noise / Offset** 等：各自有专门的 testbench 和提取定义。

这些指标的共性是：**定义里都带着“约定”**——误差带、时间窗口、参考电平、差分还是单端。自动化流程的稳健性，很大程度上取决于这些约定是否被显式、统一地固定下来。

## 5. Python 指标提取流程设计

把上面的指标拼成一个可工程化的模块，可以分成五步。

### 5.1 读取仿真结果文件

第一步是定位并打开当前 trial 的结果。提取模块的输入不是“波形对象”，而是一个 **run directory**：里面有 netlist、`spectre.out`、PSF/PSFASCII 结果目录，可能还有 OCEAN 计算出的 `performance.txt`。读取方式有两种典型路线：

- **借助结果访问接口（OCEAN/表达式）**：让仿真器侧用 `dB20`、`mag`、`phase`、`cross`、`deriv` 等表达式直接在 PSF 上计算，把标量写入文本文件，Python 只解析文本；
- **直接用 Python 解析 PSF/PSFASCII**：把波形读成数组，在 Python 里用 numpy 完成全部计算。

两条路线各有取舍。前者复用了仿真器成熟的波形算子，数值行为与 ADE 一致；后者不依赖 Cadence 进程，便于在纯 Python 环境里调试和单元测试。工程上常常混用：用表达式算复杂的环路相位，用 Python 算简单的统计量。

### 5.2 转换为统一数据结构

无论从哪条路线读入，下一步都要把数据收敛到**统一的内部表示**，屏蔽掉来源差异。一个朴素的约定是：每条信号是一对数组，外加它所属的 analysis 类型。

```python
# 仅为示意：统一的波形容器
waveform = {
    "analysis": "ac",        # ac / dc / tran
    "x": freq_array,         # 频率或时间轴
    "y": complex_response,   # 复数或实数采样
}
```

把“从哪里读”和“怎么算”解耦后，后续的指标函数就只面对统一结构，不必关心结果格式是二进制还是 ASCII。

### 5.3 执行指标提取函数

提取阶段把统一结构喂给一组**纯函数**，每个函数对应一个指标。这里推荐一种声明式的组织方式：把“需要哪些中间信号”和“报告哪些指标”写成数据，而不是散落在代码里。这样新增一个指标只是加一条声明，而不需要改动执行引擎。

```python
# 仅为示意：声明式指标定义
metric_spec = {
    "signals": {
        "ac_diff": "Voutp - Voutn",          # 差分 AC 响应
        "f_ugb":   "cross(gain_db, 0, falling)",
    },
    "metrics": {
        "gain_db":          "value(gain_db, f_low)",
        "gbw_hz":           "f_ugb",
        "phase_margin_deg": "180 + phase(ac_diff, f_ugb)",
        "power_w":          "vdd_dc * idd",
    },
}
```

这种结构的好处是：指标定义与提取引擎分离，更换电路拓扑或增删指标时，改动集中在声明上。引擎本身不需要知道“gain”“GBW”具体是什么，它只负责按声明遍历、计算、写出。

### 5.4 生成 metrics.json 或 summary.csv

提取完成后，把结果写成结构化文件。一个典型的单 trial 结果如下：

```json
{
  "gain_db": 82.5,
  "gbw_hz": 7.2e7,
  "phase_margin_deg": 64.1,
  "slew_rate_v_per_us": 128.0,
  "power_w": 0.0023,
  "valid": true
}
```

这样组织的好处很直接：

- **便于优化器读取**：optimizer 只需解析一个 JSON；
- **便于跨 trial 比较**：所有 trial 字段一致，能直接排序、筛选；
- **便于绘图**：把多个 trial 汇总成 `summary.csv` 即可画收敛曲线或散点；
- **便于筛选可行设计**：用 `valid` 与约束快速过滤；
- **便于 debug**：出问题时能立刻定位是哪个指标、哪个 trial。

### 5.5 与优化器接口对接

最后一步是把结构化 metrics 暴露给优化器。提取模块对外应当只承诺一件事：**给定一个 run directory，返回一个带有效性标记的 metrics 字典**。

```python
# 仅为示意：提取模块对外的统一入口
def extract_metrics(run_dir) -> dict:
    if not simulation_succeeded(run_dir):
        return {"valid": False, "reason": "sim_failed"}
    waves = load_waveforms(run_dir)
    metrics = run_metric_spec(waves, metric_spec)
    metrics["valid"] = all_finite(metrics)
    return metrics
```

优化器拿到的是干净的标量与一个 `valid` 标记，不需要理解 PSF、不需要知道差分定义、也不需要处理插值。提取层把所有“脏活”挡在了优化器之外。

## 6. 一个简化的指标提取示例

下面给出一段**纯示意**的 Python 代码，说明 gain 与 GBW 的提取骨架。它刻意省略了真实工程中的大量细节，只用于解释思路，不对应任何真实工程实现。

```python
import numpy as np

def extract_gain_db(freq, vout, vin):
    """低频增益（dB）。示意：单端、未做差分与收敛性检查。"""
    h = vout / vin
    gain_db = 20 * np.log10(np.abs(h[0]))
    return gain_db

def extract_gbw(freq, gain_db):
    """单位增益带宽（Hz）。示意：取首个 0 dB 穿越点并线性插值。"""
    idx = np.where(gain_db <= 0)[0]
    if len(idx) == 0:
        return None                      # 从未穿越 0 dB → 指标无效
    k = idx[0]
    return np.interp(0, [gain_db[k - 1], gain_db[k]],
                        [freq[k - 1], freq[k]])
```

这段代码足以说明“穿越点 + 插值 + 无效判断”的基本套路，但它离可用于优化闭环的提取模块还很远。真实工程中至少还要处理：

- **复数波形**：AC 结果是复数，幅值与相位要分开处理；
- **差分信号**：全差分电路要用 `Voutp − Voutn`、`Vinp − Vinn`；
- **相位展开**：PM 计算前必须正确 unwrap 相位；
- **单位换算**：dB、V/µs、mW 等统一在固定位置完成；
- **缺失结果**：结果文件不存在或信号名不匹配时优雅失败；
- **非单调曲线**：增益曲线可能多次穿越 0 dB，要选对穿越方向；
- **仿真失败**：在提取前先看 return code 与日志；
- **corner / trial / run_dir 管理**：同一指标在不同 corner、不同 trial 下要可追踪。

把这些都补齐，提取函数才能从“能算出一个数”升级为“能在大规模优化中稳定供数”。

## 7. 在模拟 IC 优化闭环中的作用

指标提取模块的真正身份，是**优化器的输入接口**。整个闭环可以写成：

```text
x_k → Spectre → waveforms_k → metrics_k → loss_k → x_{k+1}
```

其中 loss 是指标的函数：

```text
loss_k = L(metrics_k)
```

这条链路里，提取模块位于 `waveforms_k → metrics_k` 这一步。它的输出质量直接决定优化器看到的世界是否真实。如果指标提取不稳定——同一电路两次提取结果不同、把无效曲线算出一个数、单位时对时错——优化器就会被错误数据误导，朝着错误的方向采样，甚至收敛到一个仿真根本不成立的“假最优”。

因此，提取模块需要满足几条工程要求：

- **稳定**：相同输入产生相同输出；
- **可重复**：换机器、换时间不改变结果；
- **单位统一**：每个指标的单位在全流程唯一；
- **异常可识别**：无效结果被明确标记，而不是悄悄混入；
- **结果可追踪**：每个指标能回溯到具体 trial 的波形与日志；
- **与 loss function 解耦**：提取只负责“是什么”，不负责“好不好”。

最后一条尤其重要：提取层只回答“这个电路的 gain 是多少”，而“gain 多少算好、和 power 怎么权衡”属于 loss function 的职责。把这两件事分开，才能在不动提取逻辑的前提下自由调整优化目标。这正是下一篇文章的主题。

## 8. 工程实践中的注意事项

- **统一信号命名**：约定好 `voutp/voutn`、`vinp/vinn` 等命名，并在拓扑、testbench 和提取脚本之间保持一致，避免“信号名不匹配”这类最常见的提取失败。
- **全差分电路优先提取差分输出**：差分电路用单端节点提取增益会系统性偏差，应统一用差分量。
- **保存信号要提前指定**：提取需要的节点、电源电流、工作点信息，必须在 OCEAN/Spectre 的 save 列表里提前声明。没保存的信号事后无法提取——指标定义应当反向约束 save 列表。
- **不同仿真类型结果目录可能不同**：dc、ac、tran、stb 的结果可能落在不同子目录或不同 result name 下，读取时要按 analysis 取对应结果。
- **不同 PDK / ADE 配置会改变结果格式**：换 PDK、换仿真器版本、换 ADE 设置都可能影响结果格式和信号命名，流程启动时应做一次最小 smoke test 确认读取脚本仍然适用。
- **PSFASCII 大文件注意读取效率**：ASCII 结果便于调试但体积大，大规模仿真时要权衡是否落 ASCII，必要时只对需要复核的样本导出。
- **大量 trial 只保留摘要指标**：优化过程会产生海量 trial，应长期保留 `metrics.json` / `summary.csv` 这类摘要，而不是无限保留体积庞大的 raw/PSF 数据。
- **每个 trial 保留最基本的日志**：即使清理了 raw 数据，也要保留 `spectre.out` 等最小日志，便于事后定位失败原因。
- **提取函数独立于优化算法**：指标提取应当能脱离 optimizer 单独运行和测试，这样后续更换 TPE、贝叶斯优化或其他算法时，提取层无需改动。

## 9. 小结

本文讨论了模拟 IC 自动化设计中的性能指标提取层：如何认识 Spectre 输出的 PSF/PSFASCII、日志与中间结果文件，如何把离散波形抽象为数学序列，如何用明确的数学定义提取 DC gain、GBW、PM、slew rate 和 power，以及如何把这些指标组织成结构化、可追踪、带有效性标记的 metrics。

贯穿全文的一条原则是：**自动提取不只是算出一个数字，更要判断这个数字是否可信**。仿真成功、波形存在、指标有限、工作点合理，这些条件共同决定一次提取是否有效。把有效性判断前置到提取层，优化器才能始终在真实数据上工作。

```text
valid = simulation_success
        and waveform_exists
        and metric_is_finite
        and operating_point_reasonable
```

本文完成了从 Spectre 仿真波形到结构化性能指标的转换。到这里，自动化流程已经能够生成电路、运行仿真并提取指标。下一步，需要将这些指标进一步映射为优化器可以最小化的目标函数，也就是 loss function。因此，下一篇文章将介绍如何针对模拟 IC 设计构建多指标、多约束的损失函数，以及如何处理目标之间的权衡与约束的软硬边界。
